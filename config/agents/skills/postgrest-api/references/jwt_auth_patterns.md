# PostgREST JWT Authentication — Environment Quirks & Workarounds

## The Problem

The `POSTGREST_API_KEY` environment variable in this environment contains a JWT that exhibits these issues:
- **Only 2 parts** (header.payload, missing signature segment) instead of standard 3-part JWT
- **Trailing newlines** when read from file or command substitution
- **Cryptic rejection errors** from PostgREST: `JWT cryptographic operation failed`, `Expected 3 parts in JWT; got 4`, `got 2`

Despite being malformed, this JWT **does work** when passed correctly.

## Reliable Patterns

### Pattern 1: Bash with `printenv` (recommended for simple curl calls)

```bash
# Capture raw value without newlines
KEY=$(printenv POSTGREST_API_KEY)

# Use directly in curl
curl -s "http://postgrest_app:3000/rag_chunks?select=content&limit=10" \
  -H "Authorization: Bearer ${KEY}" \
  -H "Content-Type: application/json"
```

### Pattern 2: Python subprocess (recommended for complex/long-lived calls)

```python
import os
import subprocess

key = os.environ.get('POSTGREST_API_KEY', '')
auth_header = f'Authorization: Bearer {key}'

subprocess.run([
    'curl', '-s', 
    'http://postgrest_app:3000/rpc/setup_rag_schema',
    '-H', auth_header,
    '-H', 'Content-Type: application/json',
    '-d', '{}'
], check=True)
```

### Pattern 3: File-based JSON payload for RPC calls

```bash
# Write request body to file (avoids shell quoting issues with JWT in headers too)
cat > /tmp/req.json <<'EOF'
{"p_table_name": "rag_chunks", "p_embedding_column_name": "embedding"}
EOF

KEY=$(printenv POSTGREST_API_KEY)
curl -s -X POST http://postgrest_app:3000/rpc/create_rag_vector_index \
  -H "Authorization: Bearer ${KEY}" \
  -H "Content-Type: application/json" \
  -d @/tmp/req.json
```

## Anti-Patterns (Do Not Use)

| Anti-Pattern | Why It Fails |
|--------------|--------------|
| `KEY=$(cat key.txt)` | Trailing newline breaks JWT |
| `KEY=$POSTGREST_API_KEY` then `-H "Authorization: Bearer $KEY"` in double-quoted string | Shell may mangle special chars |
| `curl -H "Authorization: Bearer $(cat key.txt)"` | Trailing newline + subshell issues |
| Writing JWT to file, then `$(cat file)` | Same trailing newline problem |

## Quick Auth Verification

Before complex operations, run a lightweight test:

```bash
KEY=$(printenv POSTGREST_API_KEY)
curl -s "http://postgrest_app:3000/rag_chunks?limit=1" \
  -H "Authorization: Bearer ${KEY}" \
  | head -c 200
# Should return JSON array, not JWT error
```

## Verified Working Endpoints

All these RPCs work with the above patterns:
- `setup_rag_schema`
- `create_rag_vector_index`
- `create_vector_index`
- `find_closest_vectors`
- `deploy_function`
- `exec_sql`
- `create_table`

## Root Cause (for context)

The JWT appears to be generated by an external auth system that outputs a "short JWT" (2-part, unsigned) or the secret is not configured for verification. PostgREST in this setup is configured to accept it anyway. The exact reason is environment-specific — the workaround patterns above are what works today.
